# 02 — UX 설계 (Lockin Focus)

> 작성: UX Designer 에이전트 · 2026-04-23
> 목적: "손이 움직이기 전에 생각하게 만드는 앱"의 유저 플로우, 화면, 예외, 트레이드오프를 **구현 팀이 바로 쓸 수 있게** 정리.
> 톤: 흰색 바탕 · 미니멀 · 여백 · 짧은 문장. 압박이 아닌 **자각**.

---

## 0. 설계 철학 요약

락인 포커스의 UX는 세 겹으로 설계된다.

1. **충동(Impulse)** — 사용자가 무의식적으로 SNS 아이콘을 누른다.
2. **인지(Awareness)** — 시스템이 "잠깐" 끼어들어 사용자가 자기 행동을 본다.
3. **선택(Choice)** — 계속 진입(패널티) 또는 돌아가기(보상). **금지가 아니라 재질문.**

UI는 이 세 겹 중 **2단계(인지)**에 모든 자원을 쏟는다.
- 첫 화면은 항상 "호흡할 틈"을 준다 (10초 타이머, 1초 호흡 애니메이션).
- 버튼은 "계속" 쪽을 **작고 흐리게**, "돌아가기" 쪽을 **크고 선명하게** (유도 설계).
- 문구는 한 줄, 감정 단어 1개. 긴 설교 금지.

## 1. 유저 시나리오

### 시나리오 A — 첫 실행 / 온보딩

**트리거**
사용자가 앱을 처음 연다. 사용자는 "인스타그램을 덜 보고 싶다"는 정도의 막연한 동기만 가지고 있다.

**사용자 기대**
- 계정 가입이나 복잡한 설문 없이 5분 안에 "뭔가가 실제로 차단되는" 체감을 원한다.
- 왜 권한이 필요한지 납득하고 싶다.

**앱 반응 (4 스텝, 각 스텝 1~2 탭)**
1. **가치 제안**: "당신은 인스타를 보려던 게 아닐지도 모릅니다." + 집중 나무 일러스트.
2. **허용 앱 고르기**: "하루를 버티게 해줄 앱만 고르세요." → `FamilyActivityPicker` 호출. 사용자가 3~10개 선택.
3. **기본 스케줄**: "지금 차단을 시작할까요?" — 옵션 3개 (지금 / 평일 9-17 / 직접 설정).
4. **권한 요청**: `AuthorizationCenter.shared.requestAuthorization(for: .individual)` 호출. 성공 시 축하 화면 (나무 새싹).

**결과**
- 홈 대시보드 진입. 오늘 집중 점수 = 0, 나무 = 새싹 단계.
- 2번 스텝에서 사용자가 아무 앱도 안 골랐다면 **"기본 프리셋(전화·지도·메시지·메모)만 허용"** 제안으로 복귀 유도.

---

### 시나리오 B — 평시 홈 대시보드 진입

**트리거**
하루 중 아무 때나 사용자가 락인 포커스 앱 아이콘을 탭.

**사용자 기대**
- "내가 오늘 잘하고 있나?"를 3초 안에 알고 싶다.
- 지금 잠깐 차단 풀고 싶을 수도 있다.

**앱 반응**
- 상단: 오늘 집중 점수 (0~100) + 나무 성장 단계 (새싹/묘목/큰나무/열매).
- 중간: 오늘 앱 진입 시도 횟수 / 차단 성공 횟수 / 집중 시간.
- 하단: 주요 액션 2개 — **"집중 시작"** (대형, 색 강조) / **"잠깐 해제"** (텍스트 링크, 작게).

**결과**
- "집중 시작" 탭 → 즉시 포커스 세션(기본 25분). 세션 중엔 나무가 실시간으로 자란다.
- "잠깐 해제" 탭 → 시나리오 D의 해제 다이얼로그.

---

### 시나리오 C — 차단된 앱 실행 시도 (핵심 시나리오)

**트리거**
사용자가 홈 화면에서 Instagram 아이콘 탭. (차단 대상)

**사용자 기대**
- 평범하게 SNS를 보려 했을 뿐.
- 거부당하면 짜증이 날 것 같다.

**앱 반응 — 2가지 경로 (Architect 확인 필요, 5번 참고)**

**경로 1 (이상적): Shield → Shield Action "잠깐"** → **우리 앱 중간 인터셉트 UI가 뜸**
1. iOS Shield가 먼저 뜬다. 버튼 2개: "닫기" / "잠깐, 왜 열려 하죠?"
2. 사용자가 "잠깐" 누름 → `ShieldActionExtension`이 우리 앱 딥링크 호출 → 락인 포커스 인터셉트 화면이 뜸.
3. 인터셉트 화면에서 **10초 카운트다운** + 랜덤 개입 4종 중 1개:
   - **(V1) 10초 타이머**: 원형 progress. "10초만 기다려보세요."
   - **(V2) 이유 묻기**: "지금 왜 이 앱이 필요한가요?" 입력창 1줄. (최대 80자)
   - **(V3) 호흡 애니메이션**: 흰색 원이 4초 들숨/4초 날숨 3사이클. 말 없음.
   - **(V4) 오늘 목표**: 온보딩 때 저장한 "오늘 목표" 한 줄이 크게 뜬다.
4. 10초 경과 후 버튼 2개:
   - **"돌아가기"** (대형, 강조) → 나무 성장 +5, 홈 아이콘으로 복귀.
   - **"그래도 열기"** (작고 흐림) → 1분간 Shield 해제, 앱 실행. 에너지 -10.

**경로 2 (현실적 대안): Shield만 뜨고 그 위 버튼으로 우리 앱 열기**
- iOS Shield는 전면 차단 화면. 커스텀 UI를 그 자리에 렌더할 수 없음.
- 차선: Shield의 `primaryButtonLabel`을 "잠깐 기다려요"로 바꾸고, 누르면 `ShieldActionExtension`에서 `.defer` 반환 + 우리 앱을 딥링크로 연다. 이게 Architect가 검증해야 할 핵심.

**결과**
- "돌아가기" 다수 → 나무 열매 단계, 주간 리포트에 성공률 표시.
- "열기" 다수 → 나무가 시들지 않음 (죄책감 유도는 피함), 대신 "오늘 이 앱 7번 시도했어요. 괜찮아요?" 같은 **부드러운 피드백**.

---

### 시나리오 D — 해제 요청 (지연 해제 / 엄격 모드)

**트리거**
사용자가 홈 대시보드에서 "잠깐 해제" 또는 설정에서 "앱 선택 변경"을 누름.

**사용자 기대**
- 급한 일이 생겼다. 지금 당장 풀고 싶다.
- 엄격 모드를 자기가 켠 건 맞지만 후회한다.

**앱 반응 — 모드별 분기**

**일반 모드 (지연 해제)**
1. 다이얼로그: "지금 해제할까요? 10초 뒤에 풀립니다."
2. 원형 progress 10초 → (연속 해제 시 다음엔 30초, 그 다음 60초로 점증).
3. 해제 완료. 15분 카운트다운 배너 (상단 노란 띠). 15분 뒤 자동 재차단.

**엄격 모드 (Nuclear)**
1. 다이얼로그: "엄격 모드가 켜져 있어요. 정말 푸시겠어요?"
2. 30초 대기 카운트다운 (스킵 불가, 뒤로가기 차단).
3. 긴 문장 입력창: "나는 지금 집중을 포기하고 OOO 앱을 열기로 선택한다." 그대로 타이핑해야 제출 가능.
4. Face ID 재인증.
5. 모두 통과 → 5분만 해제. 이후 자동 재차단 + "다음엔 이 선택 대신 10초 타이머를 눌러보세요" 안내.

**결과**
- 지연 해제는 마찰(friction)만 추가, 결정은 사용자에게.
- 엄격 모드는 **귀찮아서 포기하는 구조**. 그 자체가 성공.

---

### 시나리오 E — 주간 리포트 확인

**트리거**
일요일 저녁 20:00 로컬 알림 "이번 주 리포트가 도착했어요." 사용자 탭.

**사용자 기대**
- 자기를 꾸짖는 느낌은 싫다.
- 그래도 "뭔가 달라진 게 있다"는 변화는 보고 싶다.

**앱 반응**
1. 커버 화면: "이번 주 집중 시간 X시간. 지난주보다 ±Y분." (숫자 1개만 크게)
2. 스와이프 1: 요일별 집중 시간 막대 그래프. 최고/최저 요일 하이라이트.
3. 스와이프 2: 차단 성공률. "10번 중 7번 돌아갔어요."
4. 스와이프 3: 가장 자주 시도한 앱 TOP 3 (Instagram, YouTube, X 등). 아이콘 + 시도 횟수.
5. 마지막: "다음 주 목표 한 줄로 적기" → 저장하면 시나리오 C의 V4 (오늘 목표)에 쓰임.

**결과**
- 주간 리포트는 **공유 버튼 없음** (친구 경쟁은 별도 모듈). 내면화에 집중.

---

### 시나리오 F — 도파민 디톡스 모드 활성화

**트리거**
사용자가 설정에서 "도파민 디톡스" 토글을 켠다.

**사용자 기대**
- 일일이 앱을 고르기 귀찮다. SNS·쇼핑만 딱 막아줬으면.

**앱 반응**
1. 화면: "오늘 하루 SNS/쇼핑만 차단할게요. 나머지는 평소대로." + 예상 차단 앱 아이콘 그리드 (가능하면).
2. 토글 ON → 카테고리 기반 차단 시도. (Category 자동분류 가능 여부는 Architect 확인 필요)
3. **가능한 경우**: `FamilyActivitySelection.categories`로 Social/Shopping 카테고리만 차단.
4. **불가능한 경우**: 온보딩 때 사용자가 분류한 "SNS 앱 리스트" / "쇼핑 앱 리스트" 사용 → 폴백으로 별도 Picker 노출.

**결과**
- 차단 중이면 홈 대시보드 상단에 노란 띠 "디톡스 진행 중 · 23:59 자동 해제".
- 해제 시 조용히 사라짐. 팡파르 금지.

## 2. 화면 목록 & 와이어프레임

총 **10개 화면** (온보딩 4 + 메인 6).

### 2.1 온보딩 Step 1 — 가치 제안

- **목표**: 앱의 존재 이유를 한 문장으로 각인.
- **필요한 정보**: 없음. 카피 + 일러스트.
- **주요 액션**: "시작하기" 버튼.
- **부가 액션**: 상단 "건너뛰기" (임시 계정 생성 없이 바로 대시보드). → **비추천**: 건너뛰면 차단이 하나도 안 깔려서 사용자가 "이 앱 뭐지?" 됨. **그래서 건너뛰기 버튼은 넣지 않는다.**

```
+------------------------------------------------------------+
|                                                            |
|                                                            |
|                      [  나무 새싹 일러스트  ]               |
|                                                            |
|                                                            |
|          당신은 인스타를 보려던 게                          |
|                아닐지도 몰라요.                             |
|                                                            |
|       충동과 선택 사이에 10초를 드립니다.                   |
|                                                            |
|                                                            |
|              +------------------------+                    |
|              |       시작하기         |                    |
|              +------------------------+                    |
|                      1 / 4 · · ·                           |
+------------------------------------------------------------+
```

### 2.2 온보딩 Step 2 — 허용 앱 선택

- **목표**: 사용자가 "정말 필요한 앱"을 3~10개 고르게 한다.
- **필요한 정보**: `FamilyActivityPicker` 출력 = `FamilyActivitySelection`.
- **주요 액션**: Picker 열기 → 선택 → 완료.
- **부가 액션**: "기본 프리셋 쓰기" (전화·지도·메시지·메모·시계·카메라).

```
+------------------------------------------------------------+
|   < 뒤로                                                   |
|                                                            |
|      하루를 버티게 해줄 앱만 고르세요.                      |
|                                                            |
|      나머지는 자동으로 쉬게 됩니다.                         |
|                                                            |
|   +----------------------------------------------------+   |
|   |   [ FamilyActivityPicker 시스템 UI가 여기 뜸 ]     |   |
|   |                                                    |   |
|   |   (우리가 꾸밀 수 없는 영역)                       |   |
|   |                                                    |   |
|   +----------------------------------------------------+   |
|                                                            |
|   현재 선택: 5개                                           |
|                                                            |
|              [ 기본 프리셋 쓰기 ]                          |
|              +------------------------+                    |
|              |         다음           |                    |
|              +------------------------+                    |
|                      2 / 4 · · ·                           |
+------------------------------------------------------------+
```

**판단**: "다음" 버튼은 화면 **하단 중앙**. 오른쪽 하단이 관행이지만, 첫 사용자에겐 중앙이 덜 위협적. 이후 화면들에선 오른쪽 하단으로 이동.

### 2.3 온보딩 Step 3 — 기본 스케줄

- **목표**: 차단이 언제 걸리는지 결정.
- **필요한 정보**: 프리셋 선택 or 직접 시간 입력.
- **주요 액션**: 3개 중 1개 탭 → 다음.
- **부가 액션**: "스케줄 없이 시작 (수동 차단만)" — 링크 텍스트.

```

|   |  (•) 평일 09:00 - 17:00                            |   |
|   +----------------------------------------------------+   |
|   +----------------------------------------------------+   |
|   |  ( ) 직접 설정                                     |   |
|   +----------------------------------------------------+   |
|                                                            |
|      스케줄 없이 시작 (수동 차단만)                         |
|                                                            |

```

### 2.4 온보딩 Step 4 — 권한 요청

- **목표**: `FamilyControls` Authorization 획득.
- **필요한 정보**: 시스템 권한 다이얼로그.
- **주요 액션**: "허용하기" 버튼 → `requestAuthorization` 호출.
- **부가 액션**: "왜 필요한가요?" 텍스트 링크 → 간단 설명 시트.

```
+------------------------------------------------------------+

|                                                            |
|                                                            |
|              [  락인 포커스 로고  ]                         |

|      마지막 한 단계.                                        |
|                                                            |
|      락인 포커스가 Apple의 스크린타임                       |
|      기능을 사용하려면 권한이 필요해요.                     |
|                                                            |
|      이 권한은 당신의 기기 안에서만 쓰입니다.                |
|                                                            |
|              +------------------------+                    |
|              |      허용하기          |                    |
|              +------------------------+                    |
|                                                            |
|              왜 필요한가요?                                 |
|                      4 / 4 · · ·                           |
+------------------------------------------------------------+
```

### 2.5 홈 대시보드

- **목표**: 3초 안에 "오늘 얼마나 집중했나"를 보여준다.
- **필요한 정보**: 오늘 집중 점수, 나무 단계, 시도/성공 횟수.
- **주요 액션**: "집중 시작" (대형, 색 강조).
- **부가 액션**: "잠깐 해제" (텍스트 링크) / 하단 탭 (대시보드 / 리포트 / 설정).

```
+------------------------------------------------------------+
|   락인 포커스                                       [ ⚙ ]  |
|                                                            |
|                    오늘의 집중                             |
|                       72 점                                |
|                                                            |
|                [    나무 일러스트 (큰나무)    ]             |
|                                                            |
|      시도 12 · 돌아감 9 · 집중 1시간 40분                   |
|                                                            |
|                                                            |
|              +------------------------+                    |
|              |     집중 시작          |                    |
|              +------------------------+                    |
|                                                            |
|                    잠깐 해제                                |
|                                                            |
|                                                            |
|   +----------+----------+----------+                       |
|   |   홈     |  리포트  |   설정   |                       |
|   +----------+----------+----------+                       |
```

## 3. 핵심 행동 우선순위 (TOP 3)

우리 앱은 **"차단"이 성공이 아니라 "사용자가 올바른 선택을 하는 것"**이 성공. 따라서 사용자가 가장 자주 수행할 3가지 행동은:

### 1위. "돌아가기" 누르기 (인터셉트 화면)
- **왜**: 이것이 앱의 유일한 존재 이유. 하루 10~50회 일어남.
- **UI 강조**: 흰 바탕에 검은색 채움 버튼. 화면 하단 중앙, 최소 56pt 높이. 햅틱 피드백 (success).
- **바로 옆 "그래도 열기"와의 대비**: 후자는 회색 텍스트 링크, 글자 2pt 작게, 밑줄 없음. 탭 영역도 좁게 (접근성 최소치 44pt는 유지).

### 2위. 홈 대시보드에서 "집중 시작" 누르기
- **왜**: 수동 포커스 세션 진입. 사용자가 "지금부터 1시간 진짜 집중" 선언하는 행위. 하루 1~5회.
- **UI 강조**: 홈 화면 중앙 하단의 유일한 대형 버튼. 첫 사용자에겐 24시간 후 아직 한 번도 안 눌렀다면 부드러운 코치마크 ("첫 집중을 시작해보세요").

### 3위. 허용 앱 재선택
- **왜**: 온보딩 때 대충 고른 사용자가 1주일 안에 재조정함. "왜 내 슬랙이 막혔지?" 같은 상황.
- **UI 강조**: 설정의 최상단 항목. 배지 카운트 "5개" 노출로 현재 상태 시각화. 진입은 2탭 이내.

**의도적으로 숨기는 행동** (UI에서 작게):
- "잠깐 해제", "엄격 모드 끄기", "데이터 초기화" — 모두 사용자가 **마찰을 느껴야 하는** 행동.

## 4. 예외 UX

### 4.1 FamilyControls 권한 거부
- `AuthorizationCenter`가 `.denied` 반환 시 홈 대시보드 대신 **"권한이 없어요" 전용 화면**.
- 카피: "락인 포커스는 권한 없이는 아무것도 할 수 없어요." (짧고 정직)
- 버튼: **"설정으로 이동"** → `UIApplication.shared.open(UIApplication.openSettingsURLString)`.
- 보조: "왜 이 권한이 필요한가요?" 토글. 한 문단 설명.
- 재시도 성공 시 온보딩 Step 2부터 재개.

### 4.2 시스템 오류 (Shield 설정 실패 / DeviceActivity 이벤트 누락)
- 카피: "잠깐 문제가 생겼어요. 차단이 제대로 걸리지 않을 수 있어요."
- 액션: "다시 시도" 버튼 (재-Shield 적용 호출). 로그는 App Group에 기록, 주간 리포트에는 숨김.
- 사용자 기대치 관리: **"iOS가 가끔 쉬는 시간을 가져요"** 같은 현실적 안내 (Screen Time API는 실제로 불안정).

### 4.3 앱 선택 없이 차단 시도
- 온보딩 Step 2에서 0개 선택 후 "다음" 누를 때:
  - 다이얼로그: "아무 앱도 안 고르면 모든 앱이 쉬게 돼요. 정말 시작할까요?"
  - 버튼: "앱 고르기" (권장) / "그대로 시작".
- "그대로 시작"을 택하면 기본 프리셋(전화·지도·메시지) 자동 적용 + 홈 화면에서 배너 "앱을 더 추가해보세요".

### 4.4 빈 상태 (첫날 / 집중 기록 0)
- 홈 점수 "—" 표기 (0 대신 대시). "오늘이 시작이에요." 서브카피.
- 나무는 씨앗 단계. 애니메이션 금지 (과한 자극).
- 주간 리포트 탭 진입 시: "아직 보여드릴 게 없어요. 이번 주가 끝나면 만나요."

### 4.5 로딩 상태 (Shield 적용 중)
- Shield 적용은 수백 ms 내 완료되지만 사용자 체감상 **스피너 금지** (불안감 유발).
- 대신 버튼 라벨만 잠깐 "적용 중..."으로 바뀌고 되돌아감. 햅틱 (selection) 한 번.
- 1.5초 초과 시에만 작은 원형 progress 노출.

### 4.6 Nuclear 모드 중 해제 시도
- 다이얼로그: "엄격 모드가 켜져 있어요." (시나리오 D 참고)
- 30초 타이머 중 **백그라운드 진입해도 타이머 유지되지 않음** — 다시 앱 열면 처음부터.
- 문장 입력에서 오타 → 제출 버튼 비활성. 정답 문장은 대·소문자·구두점 엄격 매칭. (Architect: Unicode normalization 필요 여부 확인)
- Face ID 실패 3회 → 10분 대기 후 재시도.

### 4.7 네트워크 실패 (주간 리포트 동기화 등)
- 현재 MVP는 **완전 로컬**. 네트워크 의존 없음.
- Phase 5의 친구 경쟁 기능에서만 네트워크 필요 → 해당 섹션은 실패 시 "오프라인" 배지만 표시, 나머지 UI 정상 동작.

## 5. 충돌 / 트레이드오프

### 5.1 Shield 기본 UI 한계 vs 중간 인터셉트 전략

**사실**
- iOS의 `ShieldConfigurationExtension`은 Shield 화면의 **텍스트, 제목, 버튼 라벨, 배경색, 아이콘**만 커스터마이즈 가능. 완전한 SwiftUI 뷰는 못 그린다.
- `ShieldActionExtension`은 Shield의 버튼 탭에 반응해 **`.none` / `.defer` / `.close`** 중 하나를 반환. 이 타이밍에 우리 앱을 **딥링크로 열 수 있는지**가 핵심 쟁점.

**우리 전략의 위험**
- "Shield 전에 우리 UI를 삽입"은 **사실상 불가능**. 앱이 실행되기 전에 iOS가 Shield를 그린다.
- 실행 가능한 시나리오:
  1. **Shield에서 "잠깐" 버튼 → 딥링크 → 우리 앱 인터셉트 UI** (시나리오 C 경로 1).
  2. **DeviceActivity 이벤트로 threshold 도달 순간 우리 앱을 열기** (유저가 백그라운드에서 앱에 들어오기 전에). 실현 가능성 낮음, iOS는 백그라운드에서 앱을 전면에 띄우지 못함.
- **솔직한 결론**: 경로 1만 신뢰 가능. 사용자는 **한 번의 Shield를 먼저 본 뒤 우리 인터셉트로 유도**된다. 온보딩에서 이 흐름을 미리 설명하는 것이 좋다.

**완화책**
- Shield 자체 카피를 우리 톤에 맞게 (`primaryButtonLabel: "잠깐 기다려요"`, `subtitle: "지금 왜 열려 하는지 같이 봐요."`).
- 우리 인터셉트 UI가 첫 방문에 한 번만 뜨도록 하는 건 **하지 않음** — 매번 떠야 학습 효과가 있음.

### 5.2 단계별 차단 (Progressive) UX 혼란 위험

**문제**
- Lv.1 알림 → Lv.2 반투명 → Lv.3 흑백 → Lv.4 완전 차단. 사용자는 지금 자기가 몇 단계에 있는지 **모른다**.
- Lv.2 반투명 오버레이와 Lv.3 iOS Filter(흑백)는 **시각적으로도 다른 매커니즘**. 일관성 훼손.

**완화책**
- 설정에서 단계 레벨을 **슬라이더 1-4**로 시각화 + 각 단계 미리보기 버튼.
- Lv.3 흑백은 "접근성 필터"라고 표기 (iOS의 Color Filter 설정을 우리가 토글하는 것). 사용자에게 "이 기능은 iOS 시스템 필터를 켜요"라고 명시.
- **기본값은 Lv.2 (반투명)**. 사용자가 자발적으로 올려야 Lv.3/4 진입.

### 5.3 "귀찮아서 안 누르는" UX vs 접근성

- "그래도 열기" 버튼을 작고 흐리게 만드는 건 **다크 패턴 경계**에 있음.
- 기준: 최소 탭 영역 44pt 유지, WCAG 대비 비율 4.5:1 이하로 떨어뜨리지 않음 (너무 흐리면 안 됨).
- 사용자가 **설정에서 이 UX를 끌 수 있어야** 한다 ("부드러운 모드": 버튼 대비를 동등하게). Phase 5 이슈로 기록.

### 5.4 Whitelist 방식의 "모르는 앱이 차단됨" 문제

- 시나리오: 사용자가 집 와이파이 켰더니 공유기 앱이 필요한데 허용 목록에 없어서 차단됨.
- 대응:
  - 인터셉트 화면 우측 상단에 **"이 앱 허용하기"** 작은 링크 — 탭하면 바로 해당 앱을 allowlist에 추가 (시스템 제약으로 앱 식별자 접근이 안 되면 설정 앱으로 딥링크).
  - Architect 확인 필요: Shield에서 차단된 앱의 bundle ID를 우리가 읽을 수 있는가.

## 6. Whitelist vs Blocklist 최종 UX 결론

**결론: 기본 Whitelist, 단 "도파민 디톡스 모드"는 Blocklist 기반.**

### 근거

1. **사용자 멘탈 모델**
   - 블랙리스트는 "내가 뭘 줄이고 싶은지"를 자기가 알아야 함. 대부분은 모름.
   - 화이트리스트는 "내가 뭐가 필요한지"만 생각하면 됨. 더 작은 의사결정.

2. **iOS 실제 앱 수**
   - 평균 사용자 기기에 앱 60~120개 설치됨. 블랙리스트로 하면 **5~15개를 고르는 작업**이지만 매번 "이건 막아야 하나?" 결정.
   - 화이트리스트는 **3~10개만 고르면 끝**. 결정 피로 총량 더 적음.

3. **FamilyControls API 특성**
   - `FamilyActivitySelection`은 카테고리/개별 앱 혼합 선택 가능. 화이트리스트도 블랙리스트도 구현 난이도 동일.
   - 단, **"허용 앱 외 전부 차단"**을 구현하려면 `shield.applicationCategories = .all(except: selection.applicationTokens)` 패턴이 필요. Architect 확인 필요 (5.3 참고).

### 그러나 — 현실적 권고

1. **완전 화이트리스트는 위험**. 설정·전화·지도 같은 **필수 시스템 앱이 자동 허용**되는 기본값이 반드시 필요. "기본 프리셋" 버튼 (온보딩 Step 2)이 안전판.
2. **첫 3일은 리스트 조정 유도**. 인터셉트 화면에서 "이 앱 허용" 링크로 쉽게 추가 가능해야 함.
3. **도파민 디톡스 모드는 Blocklist로**. SNS/쇼핑만 빠르게 끄고 싶은 상황에서는 "허용 앱 다시 고르기"가 과한 인지 부하.

### 사용자에게 보여주는 용어

- "차단할 앱" 금지. 항상 **"허용 앱"**으로 통일.
- 도파민 디톡스 모드에서만 "지금 쉬게 할 앱"이라는 다른 용어 사용 (Blocklist임을 암시).

## 7. Architect에게 확인 필요

다음 항목들은 UX 설계를 최종 확정하기 전에 기술 리서치 결과가 필요합니다.

1. **ShieldActionExtension → 메인 앱 딥링크**
   - `ShieldAction.defer`가 반환되는 순간 앱을 foreground로 띄울 수 있는가?
   - 아니면 사용자가 Shield 화면에서 우리 앱 아이콘을 다시 눌러야 하는가?
   - 둘 다 아니면 시나리오 C의 "중간 인터셉트" UX는 **대체 플로우**로 설계해야 함.

2. **차단된 앱의 bundle ID 식별**
   - Shield가 뜬 앱이 어떤 앱인지 우리가 읽을 수 있는가? (예: Instagram인지 YouTube인지)
   - 읽을 수 없다면 인터셉트 화면에서 "이 앱은..."이라고 구체 지칭이 불가능. 일반적 카피로 대체해야 함.

3. **카테고리 자동분류 (도파민 디톡스 모드용)**
   - `ActivityCategoryToken`으로 Social / Entertainment / Shopping 카테고리만 차단 가능한지.
   - 불가능하면 시나리오 F는 별도 카테고리 선택 UI가 필요 → 온보딩 분기 추가.

4. **Whitelist (허용 외 전부 차단) 구현**
   - `ManagedSettings`의 Shield를 "선택 앱 제외 전부"로 설정 가능한가?
   - `FamilyActivitySelection.includeEntireCategory` 같은 API로 전체를 뒤집을 수 있는가?
   - 불가능하면 Whitelist는 UX 환상(실제로는 사용자가 카테고리 단위로 고르는 꼴)이 됨.

5. **인터셉트 화면 딥링크 → 앱 실행 허용 ("그래도 열기" 1분)**
   - 사용자가 "그래도 열기"를 누르면 해당 앱의 Shield를 일시 해제해야 함.
   - `ManagedSettingsStore`에서 특정 앱만 임시 제거 + 1분 후 재추가 패턴이 실제로 동작하는가?
   - DeviceActivity 이벤트로 1분 스케줄링 가능?

6. **단계별 차단 Lv.3 (흑백 필터)**
   - iOS의 Color Filter (Accessibility)를 프로그래밍으로 토글할 수 있는가? (아마 불가)
   - 불가하면 Lv.3은 UI 상 "흑백 오버레이"만 우리가 그리는 걸로 축소. Lv.4와 차별화 약해짐.

7. **엄격 모드 30초 타이머 유지**
   - 사용자가 앱을 백그라운드로 보냈다가 돌아올 때 타이머 상태를 유지/리셋 중 어떤 게 맞는가?
   - 현재 UX는 "리셋"으로 설계했지만 Architect의 iOS 멀티태스킹 패턴 조사 필요.

8. **주간 리포트 알림**
   - 로컬 알림으로 일요일 20:00 트리거 가능? `UNUserNotificationCenter` + calendar trigger면 충분한지.

9. **App Group 데이터 공유**
   - 인터셉트 화면에서 "돌아가기" 누른 횟수 → 홈 대시보드 반영 지연 허용치는?
   - Extension ↔ 메인 앱 간 실시간 동기화 필요한지, 배치 업데이트 OK인지.

10. **시뮬레이터 한계**
    - Shield는 실기기 전용이라 디자인 QA를 어떻게 할지 — Architect와 테스트 전략 공유 필요.
